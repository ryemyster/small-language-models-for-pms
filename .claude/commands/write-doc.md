# write-doc

Checklist to apply before finishing any documentation file in this repo.

## Audience

Zero developer experience. No assumed knowledge of Python, Node, git, terminals, or ML.
Write as if explaining to a curious 12-year-old or a first-day PM.

## Checklist — run through this before marking a doc complete

- [ ] Every terminal command is preceded by a plain-English sentence explaining what it does
- [ ] Every terminal command is followed by "You should see something like:" and an example of the expected output
- [ ] Every technical term is defined inline — no linking out and hoping the reader follows
- [ ] No walls of text — paragraphs are 3 sentences max
- [ ] No jargon that isn't immediately explained (fine-tuning, tokenize, inference, etc.)
- [ ] No hype language: "revolutionary", "magic", "game-changing", "unlock", "powerful"
- [ ] No framework names (JTBD, dual-track, etc.) unless being critiqued
- [ ] The "when it doesn't work" case is covered — what an error looks like and what to do
- [ ] A reader with no prior context can follow the doc top to bottom without Googling anything

## Tone

Direct. No hedging. No filler. If something is hard, say it's hard. If something is easy, say it's easy.
Skeptical reader who will call out BS immediately.
